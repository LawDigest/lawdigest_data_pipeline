# 테스트 전략 및 검증 계획

## 1. 개요
데이터 파이프라인의 로직 이관 시 안정성을 보장하기 위한 테스트 전략을 정의합니다.

## 2. 테스트 단계

### 2.1 단위 테스트 (Unit Test)
- **대상**: `BillStageType` 로직, `DatabaseManager`의 쿼리 생성 로직 (Mocking 활용).
- **도구**: `pytest`, `unittest.mock`
- **주요 케이스**:
    - 단계 순서 비교 로직이 Java와 동일하게 동작하는가?
    - 트랜잭션 롤백이 예외 발생 시 정상적으로 호출되는가?

### 2.2 통합 테스트 (Integration Test)
- **대상**: 실제 DB(테스트 DB)에 연결하여 데이터 적재 수행.
- **환경**: Docker Compose로 띄운 MySQL 컨테이너 활용 권장.
- **검증 항목**:
    - `insert_bill` 호출 후 `Bill` 테이블에 데이터가 정확히 들어가는가?
    - 존재하지 않는 의원 ID로 저장 시도 시 적절히 처리되는가?

### 2.3 병행 운영 및 데이터 검증 (Dual Write & Verification)
실 운영 환경에서의 리스크를 최소화하기 위해 일정 기간 API 방식과 병행 운영합니다.

1.  **Dual Write**: 파이프라인에서 API 전송 로직과 직접 DB 적재 로직을 순차적으로 모두 실행.
2.  **Verification**:
    -   API를 통해 저장된 데이터(A)와 직접 적재 로직이 생성할 데이터(B)를 비교.
    -   (주의: 동일 DB에 동시에 쓰면 충돌나므로, 초기에는 **Dry-Run** 모드로 쿼리 로그만 남겨서 비교하거나, 별도 테스트 테이블에 적재하여 비교)

## 3. 롤백 시나리오
이관 후 운영 이슈 발생 시 즉시 복구하기 위한 시나리오입니다.

1.  **Feature Flag OFF**: 환경 변수(`USE_DIRECT_DB_ACCESS=False`) 변경으로 즉시 API 전송 모드로 전환.
2.  **데이터 복구**: 실패 시점 이후의 데이터 재수집(Re-fetch) 및 API 전송 수행.
